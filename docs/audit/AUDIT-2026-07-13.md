### [P3] `packages/corrigenda/src/corrigenda/formats/page/_custom.py:53` — Kept structural groups are reconstructed (single-space normalized), not preserved verbatim as documented
*contract-mismatch*

**Scénario.** Both the module docstring (line 11) and the caller's docstring (rewriter.py line 203) promise structural groups are 'preserved verbatim'. In reality strip_offset_groups rebuilds each kept group as f"{name} {{{body}}}" and single-space-joins them, so original formatting is altered whenever at least one offset group is removed (removed>=1). Concrete case: a line whose text was corrected has custom="readingOrder{index:0;} textStyle {offset:0;length:3;}" (no space after readingOrder, as some non-Transkribus PAGE exporters emit). Output becomes "readingOrder {index:0;}" — a space is INSERTED into the structural group, and any stray non-group characters between groups would be silently dropped. Standard Transkribus spacing (name space brace) round-trips byte-identically, so this only bites non-canonical input and does not affect identity corrections (guarded by the UNTOUCHED path and the removed==0 early return in rewriter.py). Hence low severity/confidence, but it is a genuine deviation from the 'verbatim' contract.

**Vérification.** Verified empirically and by tracing. strip_offset_groups (_custom.py line 53) rebuilds each kept structural group as f"{name} {{{body}}}" from the separately-captured name (group 1) and body (group 2), not from match.group(0), then single-space-joins them. Running the actual code: input 'readingOrder{index:0;} textStyle {offset:0;length:3;}' returns ('readingOrder {index:0;}', 1) — a space is inserted into the structural group, so output is NOT byte-verbatim. Stray inter-group characters are also dropped (input '...readingOrder {...} FOO structure {...} textStyle {offset...}' yields 'readingOrder {index:0;} structure {type:heading;}', FOO silently lost), because re.finditer only yields matches and " ".join discards the gaps. This is reachable: _strip_custom_offsets (rewriter.py:374/380) runs for corrected lines (UNTOUCHED path returns at line 355), the removed==0 early return (line 211) is bypassed because the textStyle offset group makes removed=1, and custom is read raw via el.get('custom') with no upstream normalization, so non-canonical exporter spacing survives to this function. The module docstring (line 11) and rewriter docstring (line 203) both promise 'preserved verbatim,' which the code contradicts. None of the REFUTED criteria hold — the code does not handle it, it is not impossible, and the reasoning is sound. Severity remains P3: it only alters the custom annotation microformat of lines that were actually corrected and only for non-canonical spacing; it never affects text content, TextLine geometry, or byte-parity on identity corrections (guarded by the UNTOUCHED early-out). Semantically the Transkribus custom values are equivalent, so real-world impact is cosmetic, but it is a genuine, reachable deviation from the stated 'verbatim' contract.
